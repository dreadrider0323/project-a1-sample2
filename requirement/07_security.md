# セキュリティ機能 要件定義

## 機能概要

システム全体に適用されるセキュリティ機能を定義する。
CSRF保護・レート制限・セキュリティヘッダー・SQLインジェクション対策・XSS対策を含む。

---

## EARS 要件

### CSRF保護

REQ-SEC-001
: システムは、暗号学的に安全な乱数生成器を使用して各ユーザーセッションのCSRFトークンを生成する。

REQ-SEC-002
: システムは、状態変更操作（POSTリクエスト）を実行するすべてのHTMLフォームにCSRFトークンを埋め込む。

REQ-SEC-003
: POSTリクエストを受信したとき、システムは「secrets.compare_digest」を使用したタイミングセーフな比較により、送信されたCSRFトークンをセッションに保存されたトークンと照合する。

REQ-SEC-004
: POSTリクエストに含まれるCSRFトークンがセッションに保存されたトークンと一致しない場合、システムはリクエストを403レスポンスで拒否する。

REQ-SEC-005
: POSTリクエストにCSRFトークンが含まれていない場合、システムはリクエストを403レスポンスで拒否する。

REQ-SEC-006
: システムは、CSRFトークンの比較において非ASCII文字が含まれていても未処理の例外を発生させずに処理する。

### レート制限

REQ-SEC-007
: ログインエンドポイント「POST /login」が同一IPアドレスから1分間に10件を超えるリクエストを受信している間、システムはレート制限エラーを返してログイン試行を拒否する。

REQ-SEC-008
: システムは、60秒後にリセットされるIPアドレスごとのカウンターを使用してレート制限を実装する。

### セキュリティヘッダー

REQ-SEC-009
: システムは、クリックジャッキング攻撃を防ぐため、すべてのHTTPレスポンスに「X-Frame-Options: DENY」を含める。

REQ-SEC-010
: システムは、許可されたコンテンツソースを制限するため、すべてのHTTPレスポンスに「Content-Security-Policy」ヘッダーを含める。

REQ-SEC-011
: システムは、ブラウザ機能へのアクセスを制限するため、すべてのHTTPレスポンスに「Permissions-Policy」ヘッダーを含める。

REQ-SEC-012
: 本番環境で動作中、システムはHTTPS専用アクセスを強制するため「HSTS（HTTP Strict Transport Security）」ヘッダーを含める。

REQ-SEC-013
: システムは、すべてのHTTPレスポンスに「X-Content-Type-Options: nosniff」を含める。

### SQLインジェクション対策

REQ-SEC-014
: システムは、SQLインジェクション攻撃を防ぐため、すべてのデータベース操作にパラメータ化クエリ（プリペアドステートメント）を使用する。

REQ-SEC-015
: システムは、ユーザー入力をクエリ文字列に直接連結してSQLクエリを構築しない。

### XSS対策

REQ-SEC-016
: システムは、クロスサイトスクリプティング（XSS）攻撃を防ぐため、すべてのJinja2テンプレートで自動HTMLエスケープを有効にする。

REQ-SEC-017
: ユーザー提供のデータがHTMLテンプレートでレンダリングされる箇所において、システムはすべての特殊HTML文字をエスケープする。

### パスワードセキュリティ

REQ-SEC-018
: システムは、パスワードを保存する前にブルートフォース攻撃に対抗するのに十分なワークファクターでbcryptを使用してハッシュ化する。

REQ-SEC-019
: システムは、プレーンテキストのパスワードを保存またはログに記録しない。

REQ-SEC-020
: システムは、タイミング攻撃を防ぐため、パスワード照合にタイミングセーフな比較を使用する。

### セッションセキュリティ

REQ-SEC-021
: システムは、32バイト以上の「SECRET_KEY」を使用してセッションCookieに署名する。

REQ-SEC-022
: 本番環境で動作中、システムは暗号化されていない接続での送信を防ぐため、セッションCookieに「Secure」フラグを設定する。

REQ-SEC-023
: システムは、クライアントサイドスクリプトからのアクセスを防ぐため、セッションCookieに「HttpOnly」フラグを設定する。

REQ-SEC-024
: システムは、クロスサイトナビゲーションからのCSRFリスクを軽減するため、セッションCookieに「SameSite=Lax」を設定する。

### セキュリティログ

REQ-SEC-025
: システムは、ログイン成功イベントをユーザー名とタイムスタンプとともにログに記録する。

REQ-SEC-026
: システムは、ログイン失敗の試みを試行されたユーザー名とタイムスタンプとともにログに記録する。

REQ-SEC-027
: システムは、CSRFトークン不一致イベントをリクエストパスとタイムスタンプとともにログに記録する。

REQ-SEC-028
: システムは、ログエントリに機密データ（パスワード・セッショントークン）を含めない。

## セキュリティ対策マトリクス

| 脅威 | 対策 | 実装箇所 |
|------|------|---------|
| CSRF | ダブルサブミットCookie + `secrets.compare_digest` | `security.py` |
| SQLインジェクション | パラメータ化クエリ（psycopg2） | `database.py` |
| XSS | Jinja2自動エスケープ + CSPヘッダー | テンプレート + `security.py` |
| クリックジャッキング | `X-Frame-Options: DENY` | `security.py` |
| ブルートフォース | レート制限（slowapi, 10回/分） | `security.py` |
| セッションハイジャック | HttpOnly + Secure + SameSite Cookie | `main.py` |
| パスワード漏洩 | bcryptハッシュ + ログ除外 | `auth.py` |
| MITM（本番） | HSTS ヘッダー | `security.py` |

## 環境別セキュリティ設定

| 設定項目 | 開発環境 | 本番環境 |
|---------|---------|---------|
| HSTS ヘッダー | 無効 | 有効 |
| Cookie Secure フラグ | 無効 | 有効 |
| CSP ポリシー | 緩和 | 厳格 |
| デバッグモード | 有効 | 無効 |
